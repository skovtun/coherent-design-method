## 5. Рекомендации по подключению QA (агент Quinn)

Ниже — как эффективно организовать QA в рамках BMad: когда подключать, кто пишет тесты, кто проверяет.

### 5.1. Нужно ли подключать QA?

**Да, имеет смысл** — но точечно и по чётким точкам входа:

- **Сейчас:** E2E-план выполнен вручную/одноразово; формальных gate-файлов и секции «QA Results» в историях нет (`docs/qa/gates/` пуст).
- **Цель:** воспроизводимая оценка качества (PASS/CONCERNS/FAIL), трассировка требований к тестам и явная фиксация рисков без блокировки разработки.

QA в BMad — **консультант по качеству**, а не обязательный барьер: решения по качеству принимает команда.

### 5.2. Кто пишет тесты, кто проверяет

| Роль | Действие | Комментарий |
|------|----------|-------------|
| **Dev** | Пишет unit/integration-тесты в коде, выполняет E2E по чек-листу (как сейчас). | Ближе к коду и быстрее цикл. |
| **QA (Quinn)** | Не пишет код тестов. Делает: test-design (сценарии, уровни, приоритеты), review (трассировка AC → тесты, риски), gate (PASS/CONCERNS/FAIL/WAIVED). | Фокус на стратегии и критериях «достаточно ли тестов». |
| **PM/команда** | Решает, исправлять ли CONCERNS/FAIL или принимать waiver. | QA даёт рекомендации, не блокирует релиз сам по себе. |

Итого: **тесты пишет Dev**, **проверяет полноту и критерии — QA** (через review + gate).

### 5.3. Как организовать эффективно

1. **Ввести точку «Review» в lifecycle истории**  
   Когда история в статусе Review, вызывать QA:
   - `*agent qa` → `*review {epic}.{story}` — полный разбор и обновление «QA Results» в файле истории + создание gate в `docs/qa/gates/`.
   - При необходимости: `*test-design {epic}.{story}` (сценарии до/после реализации), `*trace {epic}.{story}` (трассировка требований).

2. **Использовать gate как артефакт, а не как жёсткий lock**  
   Gate-файл (PASS/CONCERNS/FAIL/WAIVED) — вход для решения: «мержим с CONCERNS» или «сначала чиним». Не обязывать CI блокировать мерж по FAIL без согласования с командой.

3. **Epic-уровень: чек-лист + один сводный gate (опционально)**  
   Для Epic 2 уже есть E2E-план и этот отчёт. Можно один раз вызвать QA для **test-design** по Epic 2 (какие ещё сценарии/уровни желательны) и зафиксировать в `docs/qa/`. Для следующих эпиков — делать test-design в начале, review — по готовности ключевых историй.

4. **Не дублировать рутину**  
   QA не должен повторять каждый E2E-прогон вручную. Достаточно:
   - опираться на описанный E2E-план и этот отчёт;
   - при review истории проверять: «есть ли тесты/сценарии по AC» и «соответствует ли gate решению команды».

### 5.4. Конкретные шаги «подключить QA»

1. Создать каталог для gate-файлов (уже задан в `core-config.yaml`: `qa.qaLocation: docs/qa`); при первом `*review` создастся `docs/qa/gates/`.
2. По ключевым историям Epic 2 (например, 2.2, 2.4, 2.6) при переходе в Review вызвать: `*agent qa` → `*review 2.2`, `*review 2.4`, `*review 2.6` (если файлы историй есть в `devStoryLocation`).
3. Зафиксировать в процессе команды: «перед мержем в main смотрим gate; при FAIL/CONCERNS — обсуждение, при WAIVED — явная запись причины».
