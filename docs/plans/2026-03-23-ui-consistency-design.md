# UI Consistency via Reference Snippets

> Design spec for making AI-generated pages visually consistent across an entire project.

## Problem

AI generates each page independently. Even with design constraint prompts, it makes different decisions for the same UI pattern on different pages:

- Native `<select>` with `<option>` on one page, shadcn compound Select on another
- Filter icon button on Projects but not on Team
- "Status", "Priority", "Assignee" rendered as text labels instead of dropdowns
- Different toolbar compositions for semantically identical pages

Root cause: the AI **doesn't see other pages** during generation and **interprets abstract text rules** differently each time.

## Solution: Three Layers

### Layer 1 — Reference Snippets in Prompts (primary)

Replace verbose text descriptions with **concrete JSX code snippets** for every repeating UI pattern. The AI copies exact code instead of interpreting abstract rules.

**App page snippets** (injected into `DESIGN_QUALITY_APP`):

1. **Page Header** — title + description + action button
2. **Filter Toolbar** — search (flex-1) + Select dropdowns (shadcn compound pattern) + action button. No standalone filter icon buttons.
3. **Stats Grid** — KPI cards in responsive grid
4. **Data Card Grid** — project/team cards with dropdown actions
5. **Empty State** — centered icon + message + CTA button
6. **Data Table** — shadcn Table with hover rows, action column, responsive wrapper

**Auth page snippet** (injected into `DESIGN_QUALITY_AUTH`):

7. **Auth Card** — centered card with form fields, submit button, and link to other auth page

Each snippet is ~10-15 lines of JSX. Total: ~100 lines ≈ 500 tokens.

**Key rule**: snippets REPLACE the existing verbose text descriptions in `DESIGN_QUALITY_APP`, `RULES_DATA_DISPLAY`, etc. — they don't add on top. Net prompt size stays the same or decreases.

**Injection points** — snippets must be present in ALL code-generating prompts:
- Phase 5: page generation prompts in `split-generator.ts`
- Phase 4.5: shared component generation in `plan-generator.ts`
- Modification handler: single-page updates in `modification-handler.ts`
- Retry prompts: `buildLightweightPagePrompt` in `split-generator.ts`

### Layer 2 — Existing Page Context (for repeat runs)

When `coherent chat` runs on a project that **already has generated pages**:

1. Read one existing app page from disk (first `.tsx` in `app/(app)/*/page.tsx`)
2. Include its code in the prompt for new pages: "Here is an existing page in this project. Match its UI patterns exactly."
3. Falls back to Layer 1 snippets if no existing pages found

This handles pattern drift after user customizations: if the user changed the filter toolbar on Dashboard, new pages copy the customized version.

Token cost: ~2-3K per prompt. Acceptable.

**Not for first-time generation** — on first `coherent chat` run, no pages exist yet. Layer 1 snippets are sufficient.

### Layer 3 — Enhanced `formatPlanSummary`

`formatPlanSummary` currently only sends group names and shared component names to page generation prompts. `pageNotes.sections` (already generated by Phase 2) are discarded.

Change: include section names from `pageNotes` in the summary. Example:

```
ARCHITECTURE PLAN:
Groups:
  Group "app" (layout: sidebar): /dashboard, /tasks, /projects
  ...
Shared Components:
  StatCard (widget) — KPI metric card; usedBy: /dashboard, /projects
  ...
Page Sections:
  dashboard: Stats row, Recent tasks, Activity feed
  tasks: Filter toolbar, Task list
  projects: Filter toolbar, Project grid
```

This tells the AI what sections each page should have, providing structural guidance alongside the pattern snippets.

### Layer 4 — CSS Fix (already implemented)

- Added `--color-transparent: transparent` to `@theme inline` block in `tailwind-version.ts`
- Added migration check in `needsGlobalsFix` so existing projects get the fix
- Removed `variant="line"` autofix from TabsList
- Removed `border-0` autofix from TabsTrigger
- Kept: stripping AI-generated `border`/`border-input`/`outline` classes from TabsTrigger (these are AI mistakes, not shadcn defaults)

## Snippet Catalog

### 1. Page Header

```tsx
<div className="flex items-center justify-between">
  <div className="space-y-1">
    <h1 className="text-2xl font-bold tracking-tight">Page Title</h1>
    <p className="text-sm text-muted-foreground">Page description</p>
  </div>
  <Button>
    <Plus className="size-4 mr-2 shrink-0" />
    New Item
  </Button>
</div>
```

### 2. Filter Toolbar

```tsx
<div className="flex flex-wrap items-center gap-2">
  <div className="relative flex-1">
    <Search className="absolute left-3 top-1/2 -translate-y-1/2 size-4 text-muted-foreground shrink-0" />
    <Input placeholder="Search..." className="pl-9" />
  </div>
  <Select>
    <SelectTrigger className="w-[180px]">
      <SelectValue placeholder="All Status" />
    </SelectTrigger>
    <SelectContent>
      <SelectItem value="all">All Status</SelectItem>
      <SelectItem value="active">Active</SelectItem>
    </SelectContent>
  </Select>
  <Button>
    <Plus className="size-4 mr-2 shrink-0" />
    New Item
  </Button>
</div>
```

CRITICAL: Always use `<Select>` + `<SelectTrigger>` + `<SelectValue>` + `<SelectContent>` + `<SelectItem>` (shadcn compound pattern). NEVER use `<Select>` with native `<option>` elements.

Do NOT add standalone filter icon buttons (e.g., `<Button variant="outline" size="icon"><Filter /></Button>`). The Select dropdowns ARE the filters.

## Testing Strategy

- Unit tests for `formatPlanSummary` — verify pageNotes sections are included
- Unit tests for prompt construction — verify snippets appear in app/auth page prompts
- Unit tests for existing-page-context — verify file reading and inclusion
- Existing autoFixCode tests updated (TabsList/TabsTrigger rules changed)
- Manual test: run `coherent chat` on a fresh project, verify all app pages use identical filter toolbar pattern

## Non-Goals

- PRD-first architecture (deferred — simpler approach solves the problem)
- Pattern extraction from existing pages via regex (fragile, unnecessary with snippets)
- Cross-page post-generation validation (would be nice, but not needed for MVP)
- Aggressive shared component generation beyond "2+ pages" threshold
