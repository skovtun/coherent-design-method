## Why v0.14.0 needs this

v0.12.0 PR1 (apply-requests extraction) shipped with a known limitation: the 6 deterministic case bodies still live in `commands/chat/modification-handler.ts` AS WELL AS in `apply-requests/dispatch.ts`. Two copies. The temporary back-edge import (`apply-requests/dispatch-ai.ts` imports `applyModification` from `commands/chat/modification-handler.ts`) was marked "PR1 #10 will physically move the 5 AI-case bodies." That work moved out of v0.13.0 (codex consult split) and lives here.

The user-visible cost of NOT doing this: zero today. Internal cost compounds — drift surface for future contributors, layer-violation that the whole extraction was meant to fix, ~880 lines of legacy that read like the canonical path but isn't.

## Codex consult issues to pressure-test BEFORE coding

Codex on the v0.13.0 proposal already flagged these for PR2:

1. **Helper-ownership cascade:** moving `add-page` body pulls `code-generator` imports. If `code-generator` itself imports from `commands/chat/utils.ts`, the cascade extends. Audit the import graph from each AI case body 2-3 levels deep before committing to a move plan.
2. **`runPipeline` introduction:** ADR-0005 spec says chat.ts target is ~150 lines as a facade over `runPipeline()`. v0.12.0 already extracted apply-requests; runPipeline doesn't yet exist as a callable. Should v0.14.0 also create runPipeline (which absorbs pre.ts + applyRequests + post.ts), OR keep them separate calls in chat.ts?
3. **Per-commit test discipline:** the v0.12.0 PR1 pattern was 11 atomic commits with vitest green at each. v0.14.0 should mirror — but with REAL-AI smoke test BEFORE the structural moves, not after. (Item 3 from v0.13.0 was supposed to land that smoke test; deferred to user-supervised.)

## Risks specific to PR2

1. **`originalMessage` propagation.** Currently passed at chat.ts:983 (top-level loop) but NOT at chat.ts:1156 (linked-page auto-scaffold). After collapse, single site. Potential silent regression if linked-scaffold path expected `undefined` and gets the user's raw message.
2. **Side-effect ordering.** `applyManagerResult` chain in dispatch.ts vs. inline in modification-handler.ts. Parity-gate fixtures pin the deterministic types; AI-type fixtures don't exist yet (Item 3 deferred).
3. **Helper imports drag.** Each AI case body pulls 3-6 helpers from commands/chat/*. Moving them WITHOUT helper relocation creates a back-edge from apply-requests → commands/chat (re-creates the layer violation the extraction is meant to fix). Helper relocation needs to come WITH or BEFORE the case-body move.
4. **Test coverage gap.** Real-AI smoke test (Item 3) is the regression net for this whole effort. Without it, structural collapse is essentially blind. **PR2 cannot ship without Item 3 or equivalent live-AI canary.**
5. **runPipeline blast radius.** If introduced, it absorbs pre.ts + applyRequests + post.ts. Any caller of those individually breaks. v0.12.0 callers of `applyRequests` directly (chat.ts:983, appliers.ts:215) need migration.

## Open questions for codex (when this leaves spike)

1. Is the helper-ownership split (move code-generator / jsx-extractor / layout-extractor; keep request-parser / split-generator) correct? Or should helpers stay command-layer with documented one-way back-edge from apply-requests?
2. Should `runPipeline` actually be created in this release, or is it a v0.15+ concern?
3. Does the migration sequence's atomic-commit boundary work, or does the helper move (commit 4) need to happen AFTER the case-body moves (commits 6-8) to avoid intermediate broken state?
4. The 5 AI-case bodies aren't all equal complexity. `add-page` (~250 lines) + `update-page` (~280 lines) are the bulk; `link-shared` + `promote-and-link` are smaller (~80-100 each). Should we collapse the smaller ones first as warm-up, or do the largest first while context is fresh?
5. After PR2, should `apply-requests/` be renamed to something more honest (e.g., `application-pipeline/`)? The name "apply-requests" was chosen when it only handled the deterministic dispatch; post-PR2 it owns the full request lifecycle.
