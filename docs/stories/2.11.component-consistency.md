# Story 2.11: Component Consistency (Preventive + Link-to + Audit)

## Status

Review

---

## Story

**As a** developer using Coherent to build multi-page apps with shared section/widget components (e.g. PricingCard, HeroSection),  
**I want** the AI to reuse existing shared components when generating pages, and to have commands to link inline blocks to shared ones and to audit consistency,  
**so that** I avoid inline duplicates that drift from the shared version and can unify or audit when needed.

---

## Context

Shared components work for header/footer (layout type, Story 2.6). Section/widget shared components (e.g. CID-003 PricingCard) are **not** automatically reused when AI generates new pages — this leads to inline duplicates that drift from the shared version.

Three layers of defense:

- **Part A — Preventive:** At generation time, AI receives full shared component summary (all types) and is instructed to reuse first; optional warning if it still generates an inline duplicate.
- **Part B — Link-to:** User-initiated unification: replace inline with existing shared (B1), promote inline to shared and link others (B2), use shared with different props (B3).
- **Part C — Audit:** `coherent audit` reports usage verification, orphan/inline duplicates, prop consistency, unused shared.

---

## Acceptance Criteria

### Part A: Preventive Reuse (at generation time)

- [ ] **AC-A1** `coherent chat "add pricing page"` — if CID-003 (PricingCard) exists, the generated page imports it instead of creating inline cards.
- [ ] **AC-A2** Shared components summary includes **all types** (layout, section, widget) in the generation prompt for add-page (and any page generation), not only for modify-layout-block.
- [ ] **AC-A3** After generation, if output contains inline code that duplicates an existing shared component, log a warning: e.g. `⚠ Page "Pricing" contains inline code similar to CID-003 (PricingCard). Consider using the shared component instead.`

### Part B: Link-to (user-initiated unification)

- [ ] **AC-B1 (B1)** "Replace inline with existing shared": e.g. `coherent chat "on the About page, replace the hero section with CID-003"` → inline removed, import added, manifest `usedIn` updated; report e.g. "Linked About page to CID-003 (HeroSection). Inline code replaced."
- [ ] **AC-B2 (B2)** "Promote inline to shared + link others": e.g. `coherent chat "the CTA section on Home is the standard. Make it shared and use it on Pricing and About too"` → new CID created, Home/Pricing/About updated to use it; report e.g. "Created CID-005 (CTASection). Linked to 3 pages: Home, Pricing, About."
- [ ] **AC-B3 (B3)** "Use shared with different props": e.g. `coherent chat "use CID-003 on the About page but with title 'Our Story' instead of 'Features'"` → if CID-003 accepts `title`, use it; else add `title` to CID-003 and update all existing usages, then use on About; report e.g. "Added `title` prop to CID-003. Updated 3 pages."
- [ ] **AC-B4** Manifest `usedIn` is accurate after each link-to / promote operation.

### Part C: Audit (consistency report)

- [ ] **AC-C1** `coherent audit` outputs a consistency report for all shared components (usage verification, inline duplicates, unused shared, manifest mismatches).
- [ ] **AC-C2** Detects inline code structurally similar (>70% match) to an existing shared component and suggests: `coherent chat "link hero on About to CID-003"`.
- [ ] **AC-C3** Detects unused shared components (registered but not used on any page).
- [ ] **AC-C4** Detects manifest `usedIn` mismatches (claimed usage vs actual imports).
- [ ] **AC-C5** `--json` flag for machine-readable output.
- [ ] **AC-C6** Audit is informational only — does not fail the build.

## Dev Notes

- **Epic 2:** Shared components manifest `coherent.components.json`; types: layout, section, widget. Layout used in `app/layout.tsx`; section/widget imported per-page. See `docs/epic-2-shared-components.md`.
- **Chat pipeline:** `packages/cli/src/commands/chat.ts` — `parseModification`, `applyModification`; `sharedComponentsSummary` already used for modify-layout-block; extend to add-page and include all types.
- **UI-SYSTEM-PROMPT / generation:** Include instruction block:
  ```
  ## SHARED COMPONENTS (reuse first)
  Before creating any UI block, check if a matching shared component exists.
  If it does — import and use it. Do NOT recreate inline.
  Available shared components: {sharedComponentsSummary}
  When using a shared component, import from @/components/shared/{name}.
  If the shared component needs minor adaptation (e.g., different props), use its props interface — do NOT copy and modify the code inline.
  ```
- **Link-to:** B1 = replace block with import; B2 = promote (Story 2.7) + multi-page replace; B3 = add prop to shared + update usages + new usage. Update manifest `usedIn` after each.
- **Audit output format (example):**
  ```
  Coherent Design Method — Consistency Audit
  ==========================================
  Shared Components: 5 registered
  ✅ CID-001 (Header)      — layout  — used on all pages via layout.tsx
  ✅ CID-002 (Footer)      — layout  — used on all pages via layout.tsx
  ⚠  CID-003 (HeroSection) — section — used on 2 pages, but:
     → /about has a similar inline hero (78% match). Consider: coherent chat "link hero on About to CID-003"
  ⚠  CID-004 (CTASection)  — section — used on 1 page, but:
     → /pricing has a similar inline CTA (85% match). Consider: coherent chat "link CTA on Pricing to CID-004"
  ℹ  CID-005 (Testimonials) — section — registered but not used on any page
  Summary: 2 consistent, 2 with inline duplicates, 1 unused
  ```
- **Similarity (simplified):** Extract signature: top-level JSX tag names + className keywords + semantic tags. Compare token overlap; >70% → flag. Fast, deterministic, no full parser.
