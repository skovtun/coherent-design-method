# Почему вывод «говно» и как выйти на уровень Base44 / Builder.io / Lovable

**Вопрос:** Почему конкуренты по запросу делают красивые сайты, а наша платформа — минималистичную «заглушку»?

---

## 1. Диагноз (команда: Analyst + UX)

### Что видит пользователь
- Навбар: ссылки без стилей (нет hover, типографики, отступов).
- Карточки: тонкая рамка, белый фон, текст **«Card content»** — типичный плейсхолдер.
- Нет визуальной иерархии, цвета из дизайн-системы не используются, много пустого пространства.

### Почему так получается

| Причина | Где | Что не так |
|--------|-----|------------|
| **1. Жёсткий плейсхолдер в коде** | `packages/core/src/generators/PageGenerator.ts` (renderSectionComponent, card) | Если у секции нет `props.children`, генератор **всегда** подставляет `<p>Card content</p>`. Даже при правильном промпте модель не может это переопределить. |
| **2. Навбар без дизайна** | `PageGenerator.generateAppNav()` | Ссылки с классом `nav-link`; в `globals.css` для него нет стилей. Контейнер — только `flex gap-4 p-4 border-b`, без типографики и состояний (hover/focus). |
| **3. Секции без токенов** | `PageGenerator.generateSections()` | Секция: `className="mb-6"`. Нет классов из design tokens (цвета, шрифты, отступы, тени, скругления). |
| **4. Промпт не «въезжает» в генерацию** | `packages/cli/src/agents/modifier.ts` | В промпте есть принципы и ссылка на UI-SYSTEM-PROMPT.md, но: (a) полный UI-SYSTEM-PROMPT не подставляется в контекст; (b) даже если модель выдаёт богатые данные, **PageGenerator** их частично игнорирует (например, карточка без `children` → «Card content»). |

Итог: **и модель, и код-генератор** участвуют в результате. Сейчас генератор закрепляет «минимальный» вид и плейсхолдеры.

## 2. Как делают конкуренты (Architect)

- **Сильный system prompt:** полный гайд по UI/типографике/отступам в контексте модели, а не «см. файл».
- **Шаблоны страниц:** не «произвольные секции», а 2–5 готовых шаблонов (hero, grid карточек, CTA) с уже подобранными классами.
- **Нет жёстких плейсхолдеров в коде:** контент либо из ответа модели, либо явный fallback (например, section.name + «Добавьте описание»), но не «Card content».
- **Единый визуальный язык:** навбар, карточки, кнопки генерируются с одним набором классов (токены/дизайн-система).

## 3. Методология достижения высокого уровня дизайна

### Роль PM
- Зафиксировать требование: **«каждая сгенерированная страница использует design tokens и не содержит плейсхолдеров»**.
- Приоритизировать: сначала убрать «Card content» и поднять навбар; затем шаблоны и полный UI-промпт.

### Роль UX
- Описать 2–3 типа страниц (например: лендинг с hero + карточки, страница услуг, контакты) и для каждого — рекомендуемые отступы, типографика, состояния ссылок/кнопок.
- Свести это в краткий «UI checklist» для промпта и для ревью сгенерированного кода.

### Роль Architect
- **Единая точка правды:** design-system.config (токены) + globals.css (переменные) должны использоваться и в AppNav, и в PageGenerator (классы для section/card/container).
- **Шаблоны секций:** в коде или конфиге описать «типы секций» (hero, card-grid, text-block, cta) с готовыми наборами Tailwind-классов из токенов.
- **Поток данных:** модель возвращает структуру (тип секции + контент); генератор только подставляет контент в выбранный шаблон и классы, **никогда** не подставляет «Card content».

### Роль Dev
- **Убрать плейсхолдер «Card content»:** для карточки без `props.children` использовать, например, `section.name` + `section.description` или явный короткий текст вроде «Описание добавьте в конфиг», но не «Card content».
- **Стилизовать AppNav:** задать для навбара и ссылок классы из токенов (шрифт, цвет, hover, focus, отступы). Либо завести в globals.css класс `.nav-link` с этими стилями, либо генерировать Tailwind-классы в `generateAppNav()`.
- **Секции:** при генерации секций подставлять классы из токенов (например, `prose`, `space-y-4`, `rounded-lg`, `border`, `bg-card`, `p-6`) в зависимости от типа секции или layout.
- **Опционально:** пост-валидация — скрипт/чеклист, что в сгенерированных страницах нет строк типа «Card content», «Lorem», «Description here».

## 4. План действий (кратко)

| # | Действие | Ответственный | Эффект |
|---|----------|---------------|--------|
| 1 | Удалить fallback «Card content» в PageGenerator; использовать section.name / description или нейтральный текст | Dev | Исчезнет главный «мусор» на скриншоте |
| 2 | Стилизовать AppNav (Tailwind или .nav-link в globals) — hover, типографика, отступы | Dev | Навбар будет выглядеть как часть дизайна |
| 3 | Добавить в секции классы из токенов (контейнер, отступы, типографика) | Dev + Architect | Страница начнёт использовать дизайн-систему |
| 4 | Внедрить в промпт чата сжатый или полный UI-SYSTEM-PROMPT (правила без плейсхолдеров, типографика, отступы) | Dev | Модель будет чаще выдавать «богатый» контент и структуру |
| 5 | Ввести 2–3 шаблона страниц (hero, card-grid, …) с фиксированными классами; генератор выбирает шаблон по layout/типу страницы | Architect + Dev | Стабильно «красивые» страницы по запросу |

После шагов 1–2 пользователь уже увидит заметное улучшение; 3–5 выводят на уровень «красивый сайт по запросу» в духе Base44 / Builder.io / Lovable.
