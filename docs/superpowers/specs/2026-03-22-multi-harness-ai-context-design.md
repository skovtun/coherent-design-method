# Multi-Harness AI Context Generation

> Unified pipeline for generating AI editor context files from a single source of truth.

## Problem

AI context for user projects is generated by two independent modules (`cursor-rules.ts` and `claude-code.ts`) with duplicated logic: manifest/config loading, shared component formatting, and ~80% identical rules. `CLAUDE.md` drops design tokens (bug — `_config` parameter is unused). Codex, Gemini CLI, and OpenCode are unsupported. `coherent rules` does not update `.claude/commands`, `.claude/skills`, or `.claude/settings.json`.

## Solution

Replace the two independent builders with a unified pipeline: one data layer (`buildProjectContext`) feeds three thin formatters that produce `.cursorrules`, `CLAUDE.md`, and `AGENTS.md`.

## Architecture

```
design-system.config.ts + coherent.components.json
                    │
                    ▼
         buildProjectContext()          ← single data layer
         returns ProjectContext
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
  formatCursor()  formatClaude()  formatAgents()
        │           │           │
        ▼           ▼           ▼
  .cursorrules    CLAUDE.md    AGENTS.md
                  .claude/*
```

### ProjectContext

A structured object containing all project data needed by any harness:

```typescript
interface HarnessResult {
  written: boolean
  sharedCount?: number
  tokenKeys?: number
}

interface ProjectContext {
  sharedComponents: string        // full detail: import paths, props, usage examples, used-in
  sharedComponentsCompact: string // one line per component: id, name, type, file, usage count
  designTokens: string           // formatted design tokens: primary, background, foreground, etc.
  rules: string                  // all rules from "Component Rules (MANDATORY)" through "Animation"
  designQuality: string          // "Design Quality Standards" section
  architecture: string           // "Project Architecture" + "Key directories" + "Config files"
  forms: string                  // "Form Layout Rules" section
  accessibility: string          // "Accessibility (WCAG 2.2 AA)" section
  auth: string                   // "Auth Pages" section
  commands: string               // "After Making Changes" section (coherent check/fix)
  platform: string               // "Platform Overlay (DO NOT TOUCH)" section
}
```

| ProjectContext field | Source section in current template | Lines (approx) |
|---|---|---|
| `architecture` | "## Project Architecture" through "### Config files" | 88–107 |

| `rules` | "## Component Rules (MANDATORY)" through "### Animation" | 120–317 |

`buildProjectContext(manifest, config)` assembles this object once. Helper functions `buildSharedComponentsList()` and `buildDesignTokensSummary()` move into `harness-context.ts` (from `cursor-rules.ts`). The compact variant `buildSharedComponentsListCompact()` also moves here (from `claude-code.ts`).

### Formatters

Each formatter takes a `ProjectContext` and returns a string. The content is identical across harnesses — only headers and shared component format differ.

**`formatForCursor(ctx)`** → `.cursorrules`
- Header: `# Coherent Design Method — Project Rules`
- Auto-generated warning: `# Do NOT edit manually — run coherent rules to regenerate`
- Full shared component detail (`ctx.sharedComponents`)
- All other sections in order: rules, designQuality, designTokens, forms, accessibility, auth, commands, platform

**`formatForClaude(ctx)`** → `CLAUDE.md`
- Header: `# Coherent Design Method Project`
- Compact shared component list (`ctx.sharedComponentsCompact`)
- Design tokens section (fixes current bug where tokens are missing)
- All other sections identical to Cursor

**`formatForAgents(ctx)`** → `AGENTS.md`
- Header: `# Project Conventions`
- Auto-generated warning: `# Auto-generated by Coherent. Run coherent rules to regenerate.`
- Full shared component detail (`ctx.sharedComponents`, same as Cursor)
- Design tokens section
- All other sections identical to Cursor
- No tool-specific language (no references to "Cursor", "Claude Code", etc.)

### writeAllHarnessFiles()

Single entry point that replaces `writeCursorRules()`:

```typescript
interface HarnessResult {
  written: boolean
  sharedCount?: number
  tokenKeys?: number
}

async function writeAllHarnessFiles(projectRoot: string): Promise<HarnessResult> {
  let manifest: SharedComponentsManifest
  try {
    manifest = await loadManifest(projectRoot)
  } catch {
    manifest = { shared: [], nextId: 1 }
  }

  let config: DesignSystemConfig | null = null
  const configPath = join(projectRoot, 'design-system.config.ts')
  if (existsSync(configPath)) {
    try {
      const dsm = new DesignSystemManager(configPath)
      await dsm.load()
      config = dsm.getConfig()
    } catch {
      // config may be invalid during init — proceed with null
    }
  }

  const ctx = buildProjectContext(manifest, config)

  writeFileSync(join(projectRoot, '.cursorrules'), formatForCursor(ctx))
  writeFileSync(join(projectRoot, 'CLAUDE.md'), formatForClaude(ctx))
  writeFileSync(join(projectRoot, 'AGENTS.md'), formatForAgents(ctx))

  // Static .claude/* files (commands, skills, settings)
  writeClaudeCommands(projectRoot)
  writeClaudeSkills(projectRoot)
  writeClaudeSettings(projectRoot)

  const tokenKeys = config?.tokens
    ? [
        ...Object.keys(config.tokens.colors?.light ?? {}),
        ...Object.keys(config.tokens.colors?.dark ?? {}),
        ...Object.keys(config.tokens.spacing ?? {}),
        ...Object.keys(config.tokens.radius ?? {}),
      ].filter((k, i, a) => a.indexOf(k) === i).length
    : 0

  return {
    written: true,
    sharedCount: manifest.shared.length,
    tokenKeys: tokenKeys || undefined,
  }
}
```

**Note on static `.claude/*` writes:** `writeClaudeCommands`, `writeClaudeSkills`, `writeClaudeSettings` write static constants that never change between invocations. Writing them unconditionally on every call (including file-watcher) is an intentional trade-off: the files are tiny (<1KB each), and the simplicity of always writing everything outweighs the cost of conditional logic. If profiling shows this is a bottleneck, a future optimization could skip static writes when only manifest/config changed.

### regenerateAllHarnessFiles()

Wrapper for use by `coherent rules` command (the only call site without `projectRoot` in scope):

```typescript
async function regenerateAllHarnessFiles(): Promise<HarnessResult> {
  const project = findConfig()
  if (!project) {
    return { written: false }
  }
  return writeAllHarnessFiles(project.root)
}
```

## Call Site Updates

Every location that currently calls `writeCursorRules()` switches to `writeAllHarnessFiles()`. This is a **user-observable behavior change**: call sites that previously wrote only `.cursorrules` + `CLAUDE.md` will now also write `AGENTS.md` and update `.claude/*`. This is intentional — all AI context should stay in sync.

| File | Current call | New call |
|------|-------------|----------|
| `commands/init.ts` | `writeCursorRules()` + `generateClaudeCodeFiles()` | `writeAllHarnessFiles()` (replaces both calls) |
| `commands/sync.ts` | `writeCursorRules()` + `generateClaudeCodeFiles()` | `writeAllHarnessFiles()` (replaces both calls) |
| `commands/rules.ts` | `regenerateCursorRules()` | `regenerateAllHarnessFiles()` (calls `findConfig()` internally) |
| `commands/update.ts` | `writeCursorRules()` | `writeAllHarnessFiles()` |
| `commands/components.ts` | `writeCursorRules()` | `writeAllHarnessFiles()` |
| `commands/import-cmd.ts` | `writeCursorRules()` | `writeAllHarnessFiles()` |
| `commands/chat/modification-handler.ts` | `writeCursorRules()` | `writeAllHarnessFiles()` |
| `utils/file-watcher.ts` | `writeCursorRules()` | `writeAllHarnessFiles()` |

**User-facing message update:** `rules.ts` currently prints `"Updated .cursorrules and CLAUDE.md"`. Change to `"Updated .cursorrules, CLAUDE.md, and AGENTS.md"`.

## Export Integration

`commands/export.ts` must strip `AGENTS.md` in **two** places:

1. **`COPY_EXCLUDE` set** (~line 27) — add `'AGENTS.md'` to prevent copying during initial `copyDir`

## Testing

### Snapshot Regression Test

Before refactoring: capture output of current `buildCursorRules()` and `buildClaudeMdContent()` with a known manifest+config fixture. After refactoring: `formatForCursor()` must produce identical `.cursorrules`. `formatForClaude()` must produce identical `CLAUDE.md` except for the added design tokens section.

### Unit Tests

- `buildProjectContext()` — verify manifest and config are correctly transformed into all context fields
- `formatForCursor()` — verify header, full shared components, tokens, rules match current output
- `formatForClaude()` — verify header, compact shared components, tokens present (bug fix), rules
- `formatForAgents()` — verify neutral header, full shared components, tokens, rules, no tool-specific language
- `writeAllHarnessFiles()` — verify all three files + `.claude/*` are created in temp dir
- `regenerateAllHarnessFiles()` — verify returns `{ written: false }` when not in a Coherent project

### Integration Test

- Call `writeAllHarnessFiles()` on a temp project with manifest+config
- Verify `.cursorrules`, `CLAUDE.md`, `AGENTS.md`, `.claude/commands/*`, `.claude/skills/*`, `.claude/settings.json` all exist
- Verify `AGENTS.md` content includes shared components and design tokens

## Out of Scope

- **Gemini (`GEMINI.md`)**, **Windsurf (`.windsurfrules`)** — not in v1. Architecture makes adding a new harness trivial (one formatter function).
- **`.codex/` directory** — `AGENTS.md` in root is sufficient for Codex CLI.
- **Rules content changes** — existing rules text stays as-is. No rewording or restructuring.
- **Hook system** — separate concern, not part of this spec.
