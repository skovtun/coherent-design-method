# Возможности платформы и позиционирование

**Назначение документа:** опора для питчей, маркетинга и продвижения методологии Coherent Design Method. Документ **живой** — дополняйте по мере появления новых возможностей и обновления рынка.

**Как пользоваться:**
- Питчи / продажи: секции «Уникальные возможности» и «Сравнение с рынком».
- Ответы на возражения: секция «Критика, ограничения и риски».
- Обновления: при выходе нового эпика или фичи добавляйте пункты в соответствующие разделы и при необходимости обновляйте сравнение и риски.

## 1. Уникальные возможности платформы

### 1.1 Системная консистентность (design system as source of truth)

- **Один конфиг — единственный источник правды:** компоненты, токены, страницы, навигация описаны в `design-system.config.ts`. Любое изменение конфига распространяется на сгенерированный код и документацию.
- **Три уровня глобальных изменений:**
  - Токены: «сделай primary зелёным» → все страницы обновляются.
  - Компоненты: «сделай все кнопки скруглёнными» → обновляется один компонент, все использования меняются.
  - Структура: shared components (Epic 2) — правка по ID (например, CID-001) меняет Header на всех страницах.
- **Нет «амнезии» между запросами:** AI видит реестр компонентов и повторно использует их, а не создаёт дубликаты.

### 1.2 Реестр компонентов и переиспользование

- Все UI-компоненты зарегистрированы; при генерации новой страницы AI сначала проверяет реестр и переиспользует существующие компоненты.
- Отслеживание зависимостей: «где используется Button»; предупреждения при удалении или смене компонента.
- Shared Components (Epic 2): переиспользуемые блоки (header, footer, секции) с уникальными ID (CID-XXX), правка по ID/имени — изменение видно везде, где блок используется.

### 1.3 Рабочий прототип, а не макеты

- На выходе — реальное Next.js-приложение (маршруты, навигация, формы, состояние), а не статичные картинки.
- `coherent preview` — запуск dev-сервера; `coherent export` — production build, готовый к деплою (Vercel, Netlify и т.д.).

### 1.4 Встроенный Design System viewer

- Раздел `/design-system` в приложении: компоненты, варианты, размеры, токены (цвета, типографика, отступы) — генерируется автоматически.
- Секция **Shared Components**: список shared-компонентов по ID, тип, «где используется», просмотр кода (Epic 2).
- Документация и рекомендации по UX — синхронизированы с кодом (генерируются из того же конфига и правил).

### 1.5 Качество генерации и нулевые плейсхолдеры

- Ориентация на уровень shadcn/ui blocks: осмысленный контент (без «Lorem ipsum»), контекстные тексты (например, реальные тарифы на pricing-странице).
- Валидатор качества (`coherent validate`): проверки доступности, подписей, иерархии заголовков, фокуса и т.д. по заданным UX-правилам.

### 1.6 Shared Components с ID (Epic 2)

- Header, footer, повторяющиеся секции как отдельные сущности с ID (CID-001, CID-002…).
- Правка по ID или имени в чате: «в CID-001 добавь поиск» → меняется один файл, обновление на всех страницах.
- Проактивное создание: при запросе «добавь страницу с header и footer» сразу создаются shared Header и Footer, подключаются в layout.
- Команда `coherent components shared` — список shared-компонентов; дозапись DS-страниц при первом `shared add`; полная регенерация DS через `coherent ds regenerate`.

### 1.7 Инкрементальные обновления

- Изменение одного токена или компонента не требует полной перегенерации приложения — обновляются только затронутые файлы.
- Регенерация Tailwind (safelist) при изменении компонентов, чтобы стили оставались предсказуемыми.

### 1.8 Repair и валидация

- `coherent repair` — установка недостающих зависимостей, исправление типичных проблем (use client, metadata и т.д.).
- `coherent validate` — проверка страниц по качеству и UX; проверка манифеста shared components (файлы на месте).

### 1.10 Interaction Patterns (поведенческие правила)

- Правила не только про внешний вид, но и про поведение: loading states, error recovery, feedback, empty states, transitions, handoffs.
- Встроены в генерацию (AI следует им автоматически) — не ретроактивно.
- Источники: Microsoft Fluent Copilot patterns, WCAG, Nielsen Norman.
- Validation: проверки на отсутствие loading states, generic error messages, empty states без действий.

## 2. Сравнение с другими решениями на рынке

### 2.1 Сводная таблица

| Критерий | Figma | v0 / Bolt / Lovable | shadcn/ui | Storybook | Builder.io / Anima | **Coherent** |
|----------|--------|----------------------|------------|-----------|--------------------|---------------|
| Память / состояние | Нет (макеты) | Нет, каждый запрос «с нуля» | Нет генерации страниц | Нет генерации | Figma→code, без системы | **Stateful config, реестр, shared ID** |
| Переиспользование компонентов | Внутри файла | Часто дубликаты | Ручная сборка | Документация | Зависит от шаблона | **Реестр + принудительный reuse** |
| Глобальные изменения | Ручные | Регенерация страницы | Ручные | — | Ограниченно | **Токены + компоненты + shared по ID** |
| Связность многостраничного приложения | Нет | Страницы независимы | Ручная | — | Частично | **Одна конфигурация, общая навигация/layout** |
| Рабочий прототип (код) | Нет | Да, но разрозненно | Да, ручная сборка | Нет | Да | **Да, Next.js из коробки** |
| Документация дизайн-системы | Вручную | Нет | Сайт + ручная | Ручная настройка | Ограниченно | **Автогенерация /design-system** |
| Детекция дублирования (header/footer) | — | Нет | — | — | Нет | **План: AI предлагает вынести в shared (Epic 2)** |
| Поведенческие паттерны (loading, errors, empty) | Нет | Нет | Нет | — | Нет | **Встроены в генерацию + валидацию** |

### 2.2 Короткие тезисы для питча

- **Против «просто AI-генераторов»:** у них нет единого состояния и реестра — каждый запрос может породить новые кнопки и стили. Coherent хранит решение в конфиге и реестре, поэтому консистентность сохраняется.
- **Против «просто Figma»:** макеты не дают реального кода и глобальных изменений одним действием. Coherent даёт рабочий прототип и каскадные обновления.
- **Против «просто shadcn»:** shadcn — библиотека компонентов без генерации страниц и без памяти. Coherent добавляет генерацию страниц, токены и переиспользование поверх такой базы.
- **Против «просто Storybook»:** Storybook документирует компоненты, но не генерирует приложение и не связывает страницы. Coherent генерирует и приложение, и документацию из одного источника.

## 3. Критика, ограничения и риски

*Здесь собираем то, что могут сказать против платформы, и честные ограничения — чтобы готовиться к возражениям и не завышать ожидания.*

### 3.1 Зависимость от AI и провайдера

- **Критика:** качество и предсказуемость зависят от Claude/OpenAI и промптов; возможны галлюцинации или неверные правки.
- **Что делать:** явно говорить, что это «AI-assisted», не «полный автопилот»; валидация (`validate`), repair, возможность правки кода вручную; мониторинг качества генерации.

### 3.2 Сложность входа (CLI, конфиг, терминал)

- **Критика:** целевая аудитория (дизайнеры) может не любить терминал и `design-system.config.ts`.
- **Что делать:** упор на «один раз init, дальше в основном chat»; возможность IDE-интеграции; гайды и шаблоны; в перспективе — упрощённый UI поверх того же конфига.

### 3.3 Стек: только Next.js (и заявленный SPA)

- **Критика:** нет Vue, Svelte, Astro и т.д.; только определённая версия Next и React.
- **Что делать:** честно указывать в позиционировании «web, Next.js first»; расширение на другие фреймворки — как отдельная дорожная карта.

### 3.4 Figma import не в MVP

- **Критика:** в brief Figma import заявлен как must-have для релиза, но может быть отложен.
- **Что делать:** не обещать Figma в первом публичном релизе, если он не готов; формулировать «Phase 2» или «ближайшие версии».

### 3.5 Риск «перегруженного» конфига

- **Критика:** при росте проекта один большой конфиг может пугать и замедлять онбординг.
- **Что делать:** модульность конфига, примеры разбиения, документация по масштабированию.

### 3.6 Конкуренция со стороны «всё-в-одном» no-code

- **Критика:** no-code платформы дают UI без кода; здесь всё же нужны репо, терминал, базовое понимание структуры.
- **Что делать:** позиционировать как инструмент для тех, кому нужен именно код, контроль и handoff в репозиторий; преимущество — прозрачность и возможность доработки разработчиками.

### 3.7 Дублирование и детекция (Epic 2)

- **Факт:** автоматическое предложение «вынести в shared» (детекция дублирования) запланировано, но может быть сложным (AST, эвристики).
- **Что делать:** не обещать «полную автоматическую детекцию» в первой итерации; уже сейчас давать ручное создание shared по ID и проактивное создание header/footer.

## 4. Как обновлять документ

- **Новая фича или эпик:** добавить пункт в §1 (Уникальные возможности); при необходимости обновить таблицу и тезисы в §2.
- **Новый конкурент или тренд:** добавить столбец/строку в таблицу §2, скорректировать тезисы.
- **Новое возражение или инцидент:** добавить подпункт в §3, при необходимости добавить краткий «ответ» или план действий.
- **Питч-материалы:** можно копировать нужные блоки из этого файла в слайды, one-pager или скрипты продаж, сохраняя ссылку на источник.
