## 1. Goals and Background Context

### Goals

- Generated UI matches the quality of [shadcn/ui blocks](https://ui.shadcn.com/blocks) — zero placeholders, professional spacing, proper typography, meaningful content
- Every page generated by Coherent uses design tokens consistently — one change propagates everywhere
- Component-level global changes work: "make all buttons rounded" updates all instances across all pages
- The platform is reliable: `coherent init` → `coherent chat` → `coherent preview` → `coherent export` works without runtime errors
- Figma import enables designers to start from existing mockups (Phase 2 of implementation)

### Background Context

Coherent Design Method is an existing CLI platform (Phase 1 MVP complete) that generates Next.js projects with a design system. The core infrastructure works — init, chat, preview, export, Design System viewer, auto-documentation are all functional. However, the quality of generated UI is the primary blocker for release: pages look minimalistic with poor spacing, placeholder content, inconsistent styling, and navigation without design polish. The brief identifies this as the highest-priority problem.

The root cause is dual: (1) the AI modifier prompt gives principles but not constraints, so the model "invents" layouts with variable quality; (2) the code generators (PageGenerator, ComponentGenerator) have hardcoded fallbacks and minimal styling. The solution is template-based generation with strict design constraints extracted from shadcn/ui blocks, where AI fills structured content slots rather than inventing layouts from scratch.

## 2. Requirements

### Functional

- **FR1:** Every generated page MUST use design tokens from `design-system.config.ts` for colors, spacing, typography, and radii — no hardcoded pixel values or color hex codes outside tokens.
- **FR2:** Zero placeholder content in generated pages. Content must be contextual and meaningful (e.g., for a pricing page: actual tier names, features, prices — not "Title" or "Description").
- **FR3:** Every interactive element (button, link, input, card) must include hover and focus states using Tailwind utility classes.
- **FR4:** The AI modifier prompt must include concrete design constraints (spacing rules, typography hierarchy, component composition patterns) — not just principles.
- **FR5:** Page generation must use page-type templates (e.g., dashboard, pricing, contact, listing) that define proven layout structures. AI fills content slots within these templates.
- **FR6:** `coherent chat "add [page-type] page"` must produce a page visually comparable to shadcn/ui blocks of the same type.
- **FR7:** AppNav must use design tokens for styling — proper typography, hover/focus states, active state indicator, correct spacing.
- **FR8:** A post-generation validator must check generated code for forbidden patterns (placeholder strings, missing hover states, hardcoded colors) and report violations.
- **FR9:** Component-level global changes must work: modifying a component definition (e.g., border-radius, padding) must update all pages that use that component upon regeneration.
- **FR10:** `coherent repair` must fix quality issues in existing pages (add missing hover states, replace detected placeholders with contextual content).

### Non-Functional

- **NFR1:** Generation of a single page via `coherent chat` must complete in < 30 seconds (AI call + file write).
- **NFR2:** Full project regeneration (all pages, components, layout) after a token change must complete in < 60 seconds for a 20-page project.
- **NFR3:** Generated code must pass Next.js build (`next build`) without errors or warnings.
- **NFR4:** Generated code must be valid TypeScript with no type errors.
- **NFR5:** AI API token usage per page generation must stay under 50K tokens (cost control for users).
- **NFR6:** The platform must work without internet after init (except for AI API calls during `coherent chat`).

## 3. User Interface Design Goals

### Overall UX Vision

Generated applications should look and feel like they were built by a professional frontend developer using shadcn/ui. Clean, modern, with proper visual hierarchy, breathing room, and interaction feedback. The generated Design System viewer should itself be a testament to quality.

### Key Interaction Paradigms

- **Prompt-driven creation:** User describes what they want in natural language; platform generates it with consistent quality.
- **Global-by-default changes:** Token and component changes cascade to all pages automatically.
- **Preview-as-you-go:** `pnpm dev` with hot reload shows changes immediately.

### Core Screens and Views (Generated Application)

From a product perspective, these page types must be supported with high-quality templates:

1. **Dashboard** — stats cards, charts placeholder area, recent activity list
2. **Listing/Grid** — filterable grid of cards (products, services, team members)
3. **Detail page** — hero/header, content sections, related items
4. **Pricing** — tier comparison with features, CTAs
5. **Contact/Form** — form with validation states, contextual content
6. **Settings** — grouped form sections, toggles, save actions
7. **Landing/Hero** — hero section, feature grid, testimonials, CTA

### Accessibility

WCAG AA — proper contrast ratios, keyboard navigation, focus indicators, semantic HTML, ARIA labels where needed.

### Branding

Design tokens from `design-system.config.ts` drive all visual branding. Users customize through token changes, not CSS overrides.

### Target Device and Platforms

Web Responsive — all generated pages must work on mobile (320px), tablet (768px), and desktop (1280px+). Tailwind breakpoints `sm:`, `md:`, `lg:` used consistently.

## 4. Technical Assumptions

### Repository Structure: Monorepo

Existing pnpm monorepo with `@getcoherent/core` and `@getcoherent/cli`. No changes needed.

### Service Architecture

CLI tool — no backend services. AI calls are direct API calls from the CLI to Anthropic/OpenAI.

### Testing Requirements

- **Unit tests** for post-generation validator (FR8)
- **Snapshot tests** for page templates — each template type generates expected structure
- **E2E smoke test** — `coherent init` → `coherent chat "add dashboard page"` → `next build` succeeds
- Manual quality audit against shadcn/ui blocks reference for each page type

### Additional Technical Assumptions

- Full UI-SYSTEM-PROMPT.md content must be included in the AI prompt context (not just a reference to the file)
- Page templates are TypeScript functions in `packages/core/src/generators/templates/pages/` that return complete TSX strings
- AI fills content slots (titles, descriptions, item lists, feature lists) but does NOT generate layout structure — templates handle layout
- Post-generation validator runs as part of `coherent chat` pipeline and as standalone `coherent validate` command

After this epic, every page generated by `coherent chat` should look like it was hand-crafted by a professional frontend developer.

**Approach:** Template-based generation with strict design constraints. AI fills structured content slots within proven layout templates. Post-generation validation ensures no quality regressions.

### Story 1.3: Overhaul AI Modifier Prompt with Strict Constraints *(parallel with 1.2)* ✅ COMPLETE

As a **developer**,  
I want the AI modifier prompt to include the full UI-SYSTEM-PROMPT rules and structured content extraction instructions,  
so that the AI returns high-quality, structured content that fits into templates.

**Acceptance Criteria:**

1. Include the critical sections of UI-SYSTEM-PROMPT.md directly in the prompt (spacing scale, typography scale, component specs) — not just a reference to the file
2. For `add-page` requests, the prompt instructs the AI to return: (a) page type classification (dashboard, pricing, listing, contact, settings, landing, detail, generic), (b) structured content object matching the template's content schema (title, description, items, features, stats, etc.), (c) full `pageCode` as a fallback for custom page types not covered by templates

4. Add explicit required patterns: every `<button>` must have `hover:` and `focus:` classes; every `<a>` must have `hover:` class; every `<input>` must have `focus:ring-2`
5. Include 2-3 few-shot examples of correctly formatted structured content for different page types
6. Prompt token count stays under 8K tokens (balance between quality and cost)

### Story 1.4: Integrate Templates into Chat Pipeline ✅ COMPLETE

As a **developer**,  
I want `coherent chat "add [type] page"` to use page-type templates when available,  
so that generated pages always have professional layout quality.

5. Generated page passes `next build` without errors
6. A/B test: generate the same page request ("add pricing page with 3 tiers") with old approach (pure pageCode) and new approach (template + structured content) — new approach must visually match shadcn/ui blocks quality

### Story 1.6: Post-Generation Quality Validator ✅ COMPLETE

As a **developer**,  
I want a validator that checks generated code for quality issues,  
so that no page ships with placeholders, missing states, or hardcoded values.

**Acceptance Criteria:**

1. Create `packages/cli/src/utils/quality-validator.ts` with a `validatePageQuality(code: string)` function
2. Validator checks for:
   - **Placeholder content:** Strings matching "Card content", "Lorem ipsum", "Your content here", "Description", "Title" (as standalone text content, not as prop names)
   - **Missing hover states:** `<button` or `<a` elements without `hover:` class
   - **Missing focus states:** `<input` or `<textarea` or `<button` without `focus:` class
   - **Hardcoded colors:** Hex codes (#xxx) or rgb() in className that should use tokens
   - **Missing responsive:** No `md:` or `lg:` breakpoint usage in a page with grid/flex layout
3. Returns an array of `QualityIssue` objects: `{ line: number, type: string, message: string, severity: 'error' | 'warning' }`
4. Integrated into `coherent chat` pipeline: after writing a page file, validator runs and prints warnings/errors to console
5. Available as standalone `coherent validate` command that scans all `app/**/*.tsx` files
6. `coherent repair` is extended to auto-fix: add missing hover/focus states to interactive elements
